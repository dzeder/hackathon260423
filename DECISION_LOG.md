# Decision Log

Append-only. Every non-trivial fork logged here. See §19 of the spec.

Format: `HH:MM — <decision>. Reason: <why>. Alternatives considered: <list>.`

---

- `bootstrap` — Monorepo housed in the `lahore-v2` workspace rather than a fresh `~/hackathon/ohanafy-plan-demo` directory. Reason: workspace is already a git repo under Conductor; direct path to "CI/CD so others can build". Alternatives: clone into `~/hackathon/` per §4.4 verbatim (rejected — extra step, fragments git history).
- `bootstrap` — CI workflows designed for graceful fallback when secrets are absent (no `ANTHROPIC_API_KEY` → skip PR review, no `SLACK_WEBHOOK_URL` → skip ping). Reason: other engineers must be able to clone, `npm install`, and see green without any manual auth. Alternatives: hard-require secrets (rejected — blocks day-one onboarding).
- `track-b` — All four MCP servers (forecast, events, memory, network) are demo-kernel implementations backed by local JSON fixtures in `seed/`. Reason: first five hours of build must stay credentials-free so the team can iterate without waiting on CFBD, NOAA, EIA, or DynamoDB auth. Alternatives: stub servers that return empty (rejected — blocks Track A integration); wire real external APIs now (rejected — requires 5 different keys). Swap-in path is documented per-server (`loadCatalog`/`loadPeerSignals`/`MemoryStore`) so a later commit can point them at live sources.
- `track-b` — MCP tool input schemas use Zod + `zod-to-json-schema` for the MCP `ListTools` response. Reason: single source of truth for runtime validation and JSON Schema discovery; satisfies engineering-standards rule "no `any` types in Track B MCP server schemas". Alternatives: hand-written JSON Schema (rejected — drift risk with runtime validator).
- `track-b` — `ohanafy-network` enforces k-anonymity (min 3 peers per bucket, redact otherwise) in pure logic, not at the data loader. Reason: keeps the privacy invariant testable and co-located with the tool surface, not buried in fixtures. Alternatives: enforce at fixture-load time (rejected — harder to test; harder to audit).
- `track-a` — Web app holds a client-side event catalog in `src/lib/eventsCatalog.ts` and renders scenarios via pure `applyEvents` + `runThreeStatement` locally, without calling the MCP servers on each toggle. Reason: keeps the demo responsive (no RTT per click) and credentials-free. Alternatives: round-trip to MCP on every toggle (rejected — too slow for a demo; couples UI to gateway). Swap-out path: `/api/scenario` is in place so a real deployment can proxy through the MCP gateway.
- `track-a` — Copilot responses are deterministic canned replies keyed on prompt text rather than a Claude API call. Reason: demo must be reproducible and work offline during the pitch; §15 requires specific citations (CFBD, NOAA, EIA) that a live LLM might paraphrase. Alternatives: live Claude call (rejected for demo day — latency + non-determinism). Swap-out path: `/api/copilot` wraps `respond()` and can be replaced with a real Anthropic SDK call keyed on `ANTHROPIC_API_KEY` being present.
- `track-c` — `OhfyPlanMcpGateway.invokeTool` routes every call through `callout:Ohanafy_Plan_Gateway` (a Named Credential) and rejects any `toolName` not in `ALLOWED_TOOLS`. Reason: LWC must not be able to invoke arbitrary endpoints on the web-app; allow-list mirrors the MCP tool registry across all four servers plus the forecast snapshot helpers. Alternatives: pass the URL through from the LWC (rejected — trivially bypasses the allow-list); no allow-list (rejected — any future tool added server-side would be callable without a policy review).
- `track-c` — LWC `ohanafyPlanScenarioEngine` calls a single `snapshot` tool with `{scenarioId, events}` on Run, rather than chaining `apply_event` + `run_three_statement` client-side. Reason: keeps the LWC demo path deterministic (one Apex callout, one render) and lets the web-app gateway decide how to compose tools. Alternatives: multi-callout pipeline (rejected — amplifies timeout risk, harder to show a single "Scenario result" success state in the demo).
- `track-c` — LWC tests run via `sfdx-lwc-jest` behind a root-level `jest.config.js` that restricts `testMatch` to `force-app/**/__tests__/**/*.test.js` and ignores `packages/`. Reason: Jest at the monorepo root would otherwise pick up Vitest specs written in ESM/TS and fail to parse them. Alternatives: move LWC tests to a nested package (rejected — sfdx-lwc-jest insists on a root `sfdx-project.json`); use Jest for everything (rejected — Vitest already configured across Tracks A + B and we don't want to fork the harness day-one).
- `track-c` — Added minimal root `sfdx-project.json` (alongside the existing `force-app/sfdx-project.json`) so `sfdx-lwc-jest` can locate package directories from the monorepo root. Reason: tool hard-codes root discovery and won't honour a nested location. Alternatives: cd-then-run wrapper in the `test:lwc` script (rejected — obscures the project layout; breaks IDE jest runners that rely on root config).
- `track-c` — Added root `.forceignore` excluding `**/lwc/**/__tests__/**`, `jest.config.js`, and `jsconfig.json` from source deploys. Reason: without it, `sf project deploy start` tries to compile sfdx-lwc-jest test files as LWC modules and fails with `LWC1702: Invalid LWC imported identifier "createElement"` (that import is only legal in Jest specs, not in deployable components). Alternatives: move Jest tests outside the bundle (rejected — sfdx-lwc-jest convention is `__tests__/` co-located; moving them breaks the test runner); pre-build strip step (rejected — extra ceremony, easy to forget). The forceignore also covers `node_modules`, `.env*`, and `*.log` as a defence-in-depth against secrets/artefacts leaking into a deploy.
- `track-c` — `OhfyPlanMcpGateway.invokeCopilot` is a separate Apex method that hits `/api/copilot` directly, not routed through the `/api/tools/{toolName}` allow-list. Reason: the copilot is not an MCP tool — it's a scenario-aware composite endpoint in the Next.js app that internally calls `applyEvents` + `runThreeStatement` + optional live Claude. Forcing it through the tool allow-list would require either adding a synthetic `copilot` tool name (misleading — there's no MCP server for it) or splitting the LLM call across two endpoints. Alternatives: route everything through `/api/tools/` (rejected — couples the Apex surface to an architectural fiction); expose a generic "any endpoint" POST (rejected — defeats the allow-list purpose for the real MCP tools).
- `track-c` — Added a separate `ohanafyPlanCopilot` LWC rather than folding the chat into `ohanafyPlanScenarioEngine`. Reason: two app-page targets is more flexible (a CFO dashboard can show only the copilot, a planning page can show both). Also keeps each LWC bundle small and each Apex method testable in isolation. Alternatives: one combined component (rejected — forces both onto every page that wants either; larger bundle; mixed concerns in handlers). Both LWCs target `AppPage`/`RecordPage`/`HomePage`.
- `track-a` — Web app deployed to Vercel at `https://web-app-chi-puce.vercel.app` (prod alias, project `web-app` under team `daniels-projects-248be017`). `ANTHROPIC_API_KEY` set as a production env var via `vercel env add`. Reason: Salesforce-side LWCs need a public, stable URL for the Named Credential to call; `localhost` won't work from the Salesforce server. Alternatives: tunnel via ngrok (rejected — URL rotates, bad for demo day); deploy to Fly.io / Render (rejected — no MCP integration, slower to iterate). Build config: monorepo root detected, `packages/web-app` as root. No `vercel.json` — defaults work.
- `track-a` — `/api/copilot` now attempts a live Anthropic call (`claude-opus-4-7`) whenever `ANTHROPIC_API_KEY` is in `process.env`, and falls back to the canned `respond()` on any error (missing key, network failure, billing/quota error, malformed JSON, timeout). Reason: §15 demo needs live-Claude quality when credits are available, but must not break when they aren't — "absent secret = green" extends to "failed secret = graceful canned". Alternatives: always live (rejected — demo becomes non-deterministic and can flat-line if the key is revoked); always canned (rejected — leaves no swap path for a real deployment). Implementation: `src/lib/copilotLive.ts` builds a scenario-aware system prompt and asks Claude to return strict JSON (`{text, bullets, citations}`); route tags response with `source: "live"|"canned"` so the UI / telemetry can distinguish. `packages/web-app/.env.local` is a symlink to the monorepo-root `.env.local` so Next.js picks up the key without duplicating it.
- `track-a` — `/api/tools/[toolName]/route.ts` is a single dynamic dispatcher (cases: `snapshot`, `run_three_statement`, `apply_event`) rather than one Next route per tool name. Reason: `OhfyPlanMcpGateway.invokeTool` hits `callout:Ohanafy_Plan_Gateway/api/tools/{name}` for every MCP tool in the Apex allow-list; the web-app only needs to fan that out to the existing `applyEvents` + `runThreeStatement` functions, so a single handler avoids duplicating route boilerplate six ways. Alternatives: one file per tool (rejected — N routes that all call the same two functions, increases the surface a future scope cut has to prune); proxy straight to the real MCP servers (rejected — MCP servers are stdio transports, not HTTP; wrapping them on Vercel would require a separate adapter layer that isn't in scope for the demo kernel). Any tool the Apex allow-list adds later that needs different logic can branch inside the switch or peel off into its own file.
- `track-a` — Replaced single-turn stateless `/api/copilot` with a stateful ReAct orchestrator (`runCopilotTurn` in `copilotClaude.ts`) that calls Claude with tools, dispatches MCP-style tool use locally, persists turn tails, and emits structured turn logs. Reason: customer wants the in-app copilot to "feel like talking to an LLM" — single-shot didn't. Alternatives: keep canned single-turn (rejected — table stakes for production); split into separate `/api/copilot/turn` and `/api/copilot/tools` endpoints (rejected — adds round-trips for no benefit since tools are in-process).
- `track-a` — Default model is `claude-sonnet-4-6`, not `claude-opus-4-7`. Reason: ~5× cheaper at parity quality for tool-using agent work. Callers can opt into Opus per-turn via the `model` request field. Alternatives: Opus everywhere (rejected — burns budget for no real quality gain on this workload); Haiku everywhere (rejected — quality regression on multi-step tool reasoning). Haiku stays in use for the cross-conversation relevance retriever in `copilotRecall.ts`.
- `track-a` — Prompt caching wired via `cache_control: ephemeral` breakpoints on (1) static system block, (2) last tool schema, (3) last prior-turn message. Reason: turn 2+ ships ~80% fewer billable input tokens with the static persona/principles cached and the tool list (~55 schemas, ~5K tokens) cached. Alternatives: no caching (rejected — wasted ~$15/MTok on every repeated prefix); cache on every message (rejected — cache writes cost 1.25×, breakeven needs ≥1 hit, mid-message caching often misses).
- `track-a+c` — Conversation memory swapped from external Turso (libSQL) to Salesforce-native via `OhfyPlanMemoryStore` Apex REST + 3 new custom objects (`Plan_Conversation__c`, `Plan_Message__c`, `Plan_Usage_Daily__c`). Reason: customer cares about data residency — CFO conversations must stay in their org. Same Connected App powers it as the SOQL read path. Drops Turso vendor dependency entirely. Alternatives: keep Turso (rejected — vendor surface area, secrets to rotate, customer can't audit); Vercel Postgres (rejected — same residency issue); leave SQLite local file (rejected — Vercel serverless has read-only FS, doesn't survive cold starts). Tradeoff accepted: ~1-2s extra per turn from Vercel→SF round-trips, acceptable for a thoughtful CFO copilot where Anthropic latency dominates.
- `track-a+c` — Auth between Salesforce Apex gateway and Vercel `/api/copilot` is a shared-secret header (`X-Ohanafy-Client-Secret`). Apex side reads from `Ohanafy_Copilot_Config__mdt.Default.Client_Secret__c`; Vercel from `COPILOT_CLIENT_SECRET`. `COPILOT_CLIENT_SECRETS` (plural, comma-separated) supports zero-downtime rotation. Reason: simplest secure server-to-server auth; rotatable in 5 min. Alternatives: mTLS (rejected — overkill for hackathon→customer transition; client cert ops nontrivial in Vercel); JWT signed by SF (rejected — adds verification surface in the route for marginal benefit at this scale).
- `track-a` — Salesforce client uses Client Credentials OAuth 2.0 flow rather than username/password or refresh-token flow. Reason: server-to-server is the right model — the web app is acting as itself, not a named user. Run-as integration user holds the actual ACL via permission set. Alternatives: username/password (rejected — being deprecated by SF; weaker security); JWT bearer (rejected — needs cert management; client_credentials is simpler for a single-tenant per Vercel deploy).
- `track-a` — Cost guardrails: 200 turns/day, $25/day, $0.30/turn enforced server-side. Per-(customer, user, day) counters in `Plan_Usage_Daily__c` with atomic UPSERT via the `Key__c` external id. Reason: Claude can spiral on tool loops; CFO data + a CFO budget mean we can't tolerate a runaway loop charging the customer's bill. Alternatives: client-side rate limiting (rejected — bypassable); time-based circuit breaker (rejected — doesn't cover cost dimension); no caps (rejected — one bug = $1000s).
- `track-c` — `OhfyPlanMcpGateway.invokeCopilot` now takes a 4th arg `conversationId` and threads `UserInfo.getUserId()` into the POST body. Reason: per-user memory needs the SF user id; conversation continuity needs the thread id. Backwards-compat: pass `null` for conversationId to fall back to soft-rollover.
- `track-c` — LWC `ohanafyPlanCopilot` upgraded to a stateful thread UI (history load on open, message-list rendering, "New chat" button) matching the web `CopilotPanel`. Reason: both surfaces hit the same `/api/copilot` endpoint with the same memory, so they should feel the same. Three new `@AuraEnabled` methods on the gateway: `loadActiveThread`, `startNewThread`, `listThreads`.
- `track-c` — `OhfyPlanSoqlReader` Apex REST endpoint at `/services/apexrest/plan/soql` is the SOQL execution path the web-app's `query_salesforce` tool calls back into. SELECT-only, allowlisted objects (Account/Contact/Opportunity/Order/User + ohfy__* prefix), 200-row hard cap, runs `with sharing`. Reason: keeps SOQL inside the SF security model with native audit; web side never sees user passwords or arbitrary query power. Alternatives: jsforce + standard /query endpoint (rejected — bypasses Apex `with sharing` patterns and the customer's existing audit infra); stub forever (rejected — defeats the "informed" promise of the copilot).
- 20:59 — Added `@jsforce/jsforce-node` runtime dep in new workspace `packages/sf-client/`; web-app now reads forecast baseline from `ohfy__Invoice__c` aggregates instead of seed JSON. Reason: user mandate that the demo "connect to the Ohanafy database it is installed into". Sandbox `ohanafy-hack-sandbox` was empty so we also wrote `scripts/load-fixtures-to-org.py` to bulk-insert 6 retailer accounts + 36 invoices + 36 depletions whose monthly rollups exactly match `seed/baseline-forecast.json`. Alternatives considered: keep seed-only and stub the connection (rejected — defeats the user's directive); load fixtures into NEW custom objects instead of `ohfy__` (rejected — never exercises the real schema). Why `jsforce-node` over the legacy `jsforce` package: ESM-native, Node-only, smaller bundle, actively maintained at the `@jsforce` org; the legacy CJS `jsforce` is in maintenance mode.
- 20:59 — Auth picks SFDX auth URL (`SF_AUTH_URL` env) over `SF_LOGIN`/`SF_PW`/`SF_URL`. Reason: same mechanism CI already uses (`.github/workflows/ci.yml` with `SF_AUTH_URL` GitHub secret + `sf org login sfdx-url`); refresh-token rotation > password+token; one env var instead of three. Extracted via `sf org display --target-org ohanafy-hack-sandbox --json --verbose` and stored in `.env.local`. Alternatives considered: jsforce username+password (rejected — needs MFA-bypass security token, brittle); JWT bearer flow with Connected App (rejected — requires creating a Connected App in the org, more setup than the slice needed). The legacy `SF_LOGIN`/`SF_PW`/`SF_URL` vars in `.env.local.example` will be removed in a follow-up so we have one canonical auth path.
- 20:59 — Two read paths to `ohfy__` data, on purpose. JS/TS path (web-app + MCP servers): `@ohanafy-plan/sf-client` via jsforce + `SF_AUTH_URL`. Apex/LWC path (Track C, deferred): direct `@AuraEnabled` SOQL with `WITH USER_MODE`. Reason: LWCs are already in-org — sending a Vercel→jsforce→SF→Vercel→LWC round-trip when the LWC could query directly is silly; in-org SOQL respects FLS/CRUD via `WITH USER_MODE` and avoids callout governor limits. Alternatives considered: route everything through web-app (rejected — adds latency and a public surface for in-org data); single jsforce client used by both LWC (via Lightning fetch) and Vercel (rejected — LWC can't run jsforce; would require a server-side proxy that we'd then immediately collapse).
- 21:17 — `ohanafy-forecast` MCP server's `loadBaseline()` now reads from `ohfy__Invoice__c` aggregates via `@ohanafy-plan/sf-client` when `SF_AUTH_URL` is set, with a 5-min in-process cache. Reason: keeps the MCP path consistent with the web-app — both compute scenarios off the same org-derived baseline rather than the web-app being live and the MCP being stubbed. Cache is per-process so back-to-back tool calls in a single Claude turn (apply_event → run_three_statement → snapshot) make at most one SOQL round-trip. Alternatives considered: no cache (rejected — three SF round-trips per copilot turn is wasteful given the data is identical); always read from org with no fallback (rejected — would fail-closed in dev/CI without sandbox creds, breaking the "absent secret = green" rule); fold the conversion into sf-client as a shared `getForecastBaseline()` helper to DRY against `salesforceDataSource.ts` (deferred — duplication is small and the slice goal is wiring, not refactor). Events catalog + memory store / knowledge base remain on seed JSON until Track C lands the supporting `Plan_Event_Template__c` / `Plan_Knowledge_Article__c` / `Plan_Scenario_Decision__c` custom objects.
