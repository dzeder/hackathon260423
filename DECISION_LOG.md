# Decision Log

Append-only. Every non-trivial fork logged here. See §19 of the spec.

Format: `HH:MM — <decision>. Reason: <why>. Alternatives considered: <list>.`

---

- `bootstrap` — Monorepo housed in the `lahore-v2` workspace rather than a fresh `~/hackathon/ohanafy-plan-demo` directory. Reason: workspace is already a git repo under Conductor; direct path to "CI/CD so others can build". Alternatives: clone into `~/hackathon/` per §4.4 verbatim (rejected — extra step, fragments git history).
- `bootstrap` — CI workflows designed for graceful fallback when secrets are absent (no `ANTHROPIC_API_KEY` → skip PR review, no `SLACK_WEBHOOK_URL` → skip ping). Reason: other engineers must be able to clone, `npm install`, and see green without any manual auth. Alternatives: hard-require secrets (rejected — blocks day-one onboarding).
- `track-b` — All four MCP servers (forecast, events, memory, network) are demo-kernel implementations backed by local JSON fixtures in `seed/`. Reason: first five hours of build must stay credentials-free so the team can iterate without waiting on CFBD, NOAA, EIA, or DynamoDB auth. Alternatives: stub servers that return empty (rejected — blocks Track A integration); wire real external APIs now (rejected — requires 5 different keys). Swap-in path is documented per-server (`loadCatalog`/`loadPeerSignals`/`MemoryStore`) so a later commit can point them at live sources.
- `track-b` — MCP tool input schemas use Zod + `zod-to-json-schema` for the MCP `ListTools` response. Reason: single source of truth for runtime validation and JSON Schema discovery; satisfies engineering-standards rule "no `any` types in Track B MCP server schemas". Alternatives: hand-written JSON Schema (rejected — drift risk with runtime validator).
- `track-b` — `ohanafy-network` enforces k-anonymity (min 3 peers per bucket, redact otherwise) in pure logic, not at the data loader. Reason: keeps the privacy invariant testable and co-located with the tool surface, not buried in fixtures. Alternatives: enforce at fixture-load time (rejected — harder to test; harder to audit).
- `track-a` — Web app holds a client-side event catalog in `src/lib/eventsCatalog.ts` and renders scenarios via pure `applyEvents` + `runThreeStatement` locally, without calling the MCP servers on each toggle. Reason: keeps the demo responsive (no RTT per click) and credentials-free. Alternatives: round-trip to MCP on every toggle (rejected — too slow for a demo; couples UI to gateway). Swap-out path: `/api/scenario` is in place so a real deployment can proxy through the MCP gateway.
- `track-a` — Copilot responses are deterministic canned replies keyed on prompt text rather than a Claude API call. Reason: demo must be reproducible and work offline during the pitch; §15 requires specific citations (CFBD, NOAA, EIA) that a live LLM might paraphrase. Alternatives: live Claude call (rejected for demo day — latency + non-determinism). Swap-out path: `/api/copilot` wraps `respond()` and can be replaced with a real Anthropic SDK call keyed on `ANTHROPIC_API_KEY` being present.
